A Practical Note on Context Windows

Context windows are usually described as a single number: one million tokens, two hundred thousand tokens, or some other large limit. That number is useful, but it hides the fact that tokens are not words, characters, or bytes. A dense code file, a transcript full of short turns, and an essay with long paragraphs can consume context at different rates. The same string of text can also consume different amounts of context across providers, because each provider runs the text through its own tokenizer before counting.

For people building with language models, this matters in mundane ways. A retrieval system might include five documents when it should include three. A coding assistant might fit the source file but not the surrounding tests. A research workflow might appear stable until a PDF extraction step introduces odd punctuation, page headers, or repeated footers. None of these failures are dramatic in isolation. They show up as small drops in answer quality, small increases in cost, or small surprises at the end of a billing period.

The safest approach is to measure the actual input shape that will be sent to the model. For cross-model comparisons, measure the same controlled text with each provider's counting surface. For production systems, also measure the full request payload, because system prompts, tool schemas, and wrapper formats can add meaningful overhead. A useful question to ask of any pipeline is: of the tokens we are paying for, how many are in the user's content and how many are in the wrapping that delivers it to the model? The answer is often surprising and is rarely the same across providers.

A related caution is that context-window numbers are advertised as ceilings, but most production systems cannot push to the ceiling without trouble. Long inputs raise the chance of partial attention, increase latency, and can interact poorly with safety filters that look at the entire prompt. The headline number is a useful upper bound, but the practical limit is usually somewhat lower, and the practical limit depends on the workload. A summarization workflow may run comfortably at most of the advertised window. A multi-tool agent may start to hit invisible internal limits long before the documented one.

Tokenization itself is rarely a topic the average user thinks about, and there is no reason it should be. The technology has been engineered so that most users can ignore it most of the time. The cases where it leaks out are usually adjacent to other concerns: the document does not fit, the cost was higher than expected, the same prompt produced a longer reply on a different provider. In each case, tokenization is at least partly responsible, and the fix usually starts with measuring before reasoning.

The result is less glamorous than a leaderboard, but more operationally useful: a small table showing what each model says the same text costs to read. Such a table will not tell anyone which model is best, and it will not settle any vendor argument. It will tell a team whether a planned migration is cheap or expensive in tokens, and it will catch a tokenizer change between model revisions before that change shows up in production.

For readers who want a single takeaway: token counts are a property of the model, not the text. Two systems can read identical paragraphs and report different numbers, and both systems can be entirely truthful. The right response is not skepticism of either provider; it is to measure the workloads you actually care about, on the providers you are actually considering, and to treat the resulting numbers as data points rather than universal truths. The measurement itself, with the count endpoints both major providers now expose, is a small effort. The habit of doing it before drawing conclusions is the harder part.

A final note on the tone of this kind of work. Tokenization comparisons read, at first, like an exercise in pedantry. Two providers count slightly differently; the differences are usually small; the practical impact is bounded. None of those observations are wrong, and yet the work matters anyway. Small differences accumulate over millions of requests. Bounded effects can still be the deciding factor in a tight bid. And, perhaps most importantly, public habits of measurement push providers toward documentation, toward stable behavior, and toward useful counting endpoints. The article that quotes a token-count comparison politely is, in a small way, raising the floor for everyone who depends on these systems.
